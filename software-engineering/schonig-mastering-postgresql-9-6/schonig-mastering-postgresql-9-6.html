3.4.2 Windowing queries

  1. Partitioning data
    • the number of rows returned doesn't change (unlike with GROUP BY)
    • ordering within a partition matters for aggregation 
    SELECT country,
       year,
       production,
       consumption,
       avg(production) OVER (PARTITION BY country)
FROM t_oil;

    SELECT country,
       year,
       production,
       min(production) OVER (),
       min(production) OVER (ORDER BY year)
FROM t_oil
WHERE year BETWEEN 1978 AND 1983
      AND country = 'Iran';
  2. Using sliding windows

    This is a clear query to show the results of sliding windows. 

    SELECT *,
       array_agg(id) OVER (ORDER BY id ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING)
FROM generate_series(1, 5) as id;
    • Can be unbounded on both sides by UNBOUNDED FOLLOWING, or UNBOUNDED PRECEDING
  3. Abstracting window clauses
    SELECT country,
       year,
       production,
       min(production) OVER (w),
       max(production) OVER (w)
FROM t_oil
WHERE country = 'Canada' AND year BETWEEN 1980 AND 1985
WINDOW w AS (ORDER BY year)
    country year production min max
    Canada 1980 1764 1764 1764
    Canada 1981 1610 1610 1764
    Canada 1982 1590 1590 1764
    Canada 1983 1661 1590 1764
    Canada 1984 1775 1590 1775
    Canada 1985 1812 1590 1812

    • rank

3.4.3 Writing your own aggregates

CREATE FUNCTION taxi_per_line (numeric, numeric)
RETURN numeric AS
$$
BEGIN
RAISE NOTICE 'intermediate: %, per row: %', $1, $2;
RETURN $1 + $2 * 2.2;
END;
$$ LANGUAGE 'plpgsql';

CREATE AGGREGATE taxi_price (numeric)
(
    INITCOND = 2.5,
    SFUNC = taxi_per_line,
    STYPE = numeric
);

3.6.1 Optimization strategies

  • constant folding
  • view inlining
  • join reordering
  • flattening subselects
  • join pruning
  • applying equality constraints
  • function inlining
  • distribute over set operations (UNION [ALL], etc.)

It is not difficult to make the process fail (e.g. by specifying OFFSET = 0). Always run explain on a query.

3.6.3 Relevant system catalogs

pgclass catalogs tables and most everything else that has columns or is otherwise similar to a table.

  • pgstats is a view on top of pgstatistics
  • pgstatistics stores statistical data about the contents of the database
  • pgstatusertables contains one row for each table in the current database, showing statistics about accesses to that specific table

3.6.4 Node types in a query plan

  1. Scans
    1. Index-Only Scan
      • Only needs to fetch index pages
      • Requires data to be fetched to be available from the index
      • MVCC visibility information is not stored in the index, but the table's visibility map has a flag for each heap page that indicates when an entire page is old enough to be visible to all current and future transactions. (see 2, Chapter 11.11 Index-Only Scans)
    2. Index Scan
      • Rows are fetched in index order from the index, and then separately retrieved from the heap
    3. Seq Scan
      • Entire table is scanned
    4. Bitmap Heap Scan
      • Used after a Bitmap Index Scan, retrieves the pages selected by the Bitmap Index Scan
      • Needs to apply the filter condition again, since rows in the heap page fetched might not satisfy it.
    5. Bitmap Index Scan
      • Gathers the pages of the rows to retrieved from the index
    6. Function Scan
  2. Joins
    1. Hash Join
      • The rows of one of the tables are collected in a hash table (which one is indicated by Hash)
      • These rows are then looked up from the row set of the other table
    2. Merge Join
      • Requires the tables to be sorted on fields in the join condition
      • Merging then takes time proportional to the number sum of the rows of the tables to merge.
    3. Nested Loop
      • A nested loop takes time proportional to the products of the number of rows to merge.
  3. Miscellaneous
    1. Append
      • Appends to result sets
    2. Unique
      • Filter out duplicates
      • Can be expensive (see 3, p. 163)
    3. Sort
      • Sort the result set
      1. external sort Disk
      2. quicksort Memory
      3. top-N heapsort Memory
        • To only provide top-n rows
    4. Limit
      • Limits the result set
    5. Subquery Scan
  4. Aggregates
    1. HashAggregate
      • Aggregate by building an in-memory hash table
    2. GroupAggregate
      • Requires sorted data
      • Takes linear time, but can emit partial results

3.6.5 Understanding execution plans: Spotting problems

  • Start where the query times jump
  • Inspect estimates
    • Maybe row sizes are over- or under-estimated due to wrong statistics
    • Maybe cross-column correlations make the estimates off (statistics in PostgreSQL 9.6 are univariate).
  • Inspect buffer usage

3.6.6 Miscellaneous notes

  1. CLUSTER clauses
    • Rewrite the table in the same order as a (B-tree) index (3, p. 170). Requires a table lock.
  2. Inner joins may be reordered
    • Outer joins cannot always be reordered
    • This is probably a restatement of the algebraic properties of both of these types of joins in the relational algebra.
  3. GROUP BY 1
    • It is possible to specify only the indices of the column to group or order by.

3.6.7 Partitioning data

  1. Modifying inherited structure
    • Adding and removing columns propagates to the child tables
    • Adding indexes does not
    • It is also simple to change the parent of the child table. Maybe for moving data from active to history.

3.6.8 Adjusting parameters

  1. workmem
    • Query plans obviously depend on working memory.
  2. maintenanceworkmem
    • Memory available for maintenance work (creating indices, etc.). Not so useful, maybe for creating indices on the fly.

3.7 DONE 7. Writing stored procedures

3.7.1 Takeaways

  • Probably better to use the jv8 extension that allows for using JavaScript in PostgreSQL as a trusted language. Also pglpsql seems quite simple.
  1. Triggers are useful and flexible
    • They run in alphabetical order!
  2. Types of functions
    • volatile: no assumptions on return value
    • stable: referentially transparent within a transaction
    • immutable: referentially transparent
  3. PL/pgSQL is simple and takes care of more things
    • For example, caching execution plans (see 3, p.228).
  4. Can create your own operators, type casts, and even collations
    • Collation is combining data, but I think refers mostly to sort orders in this context.
